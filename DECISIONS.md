## Решения

### DEC-001: User-Agent — гибридный подход
- **Дата:** 2026-04-05
- **Контекст:** тестирование на 5 крупных сайтах показало, что `Compliance152Bot/0.1` блокируется на 4 из 5 (WB, Tinkoff, Ozon, DNS). PlaywrightCrawler запускает реальный Chromium — браузерный UA здесь честен технически.
- **Варианты:**
  - (А) Честный бот-UA везде — сильно ограничивает охват, большинство крупных сайтов недоступны
  - (Б) Браузерный UA везде (в т.ч. httpx) — в httpx это бесполезный обман, WAF всё равно детектирует по отсутствию JS и fingerprint
  - **(В) Гибрид: SiteScanner — честный UA, PlaywrightCrawler — браузерный UA** ✓
- **Решение:** вариант В. SiteScanner сохраняет `Compliance152Bot/0.1`. PlaywrightCrawler использует реальный браузерный User-Agent (это не обман — он и есть настоящий Chromium).
- **Компромиссы:** часть сайтов всё равно детектирует headless-браузеры по другим признакам (canvas fingerprint, отсутствие плагинов и т.д.). Полного обхода не гарантировано.
- **Примечание для отчётов:** режим сканирования (httpx vs Playwright) пользователю не показывать — он пришёл проверить соответствие 152-ФЗ, а не разбираться в технических деталях.
- **Связанные файлы:** `src/scanner/playwright_crawler.py` (строка с `user_agent=`)

### DEC-002: Автоматический fallback SiteScanner → PlaywrightCrawler
- **Дата:** 2026-04-05
- **Контекст:** тестирование показало, что SiteScanner возвращает «плохой» результат (0 страниц, 4xx, пустой текст политики) на крупных сайтах без явной ошибки для пользователя. Ручная кнопка «попробовать в браузерном режиме» не подходит — пользователь не поймёт зачем.
- **Варианты:**
  - (А) Ручная кнопка — требует от пользователя понимания разницы краулеров, большинство уйдут с пустым результатом
  - **(Б) Автоматический fallback** ✓
- **Решение:** вариант Б. Если SiteScanner вернул «плохой» результат — автоматически перезапускать через PlaywrightCrawler. Критерии плохого результата: `pages_scanned=0`, или наличие 4xx в errors, или `privacy_policy.found=False` при `pages_scanned≥1` и `text_len<100`.
- **Компромиссы:** двойное время сканирования в худшем случае; нужна защита от бесконечного fallback (только один уровень).
- **Связанные файлы:** `src/api/server.py` (`_build_scanner`), `src/scanner/crawler.py`, `src/scanner/playwright_crawler.py`

### DEC-003: el-ed.ru — первый эталонный сайт для валидации сканера
- **Дата:** 2026-04-05
- **Контекст:** после тестирования на 5 крупных сайтах (WB, Tinkoff, Ozon, mos.ru, DNS) стало ясно, что WAF и bot-fingerprinting делают их неподходящими для MVP-валидации сканера. Нужен набор сайтов без агрессивной защиты, представляющих реальную целевую аудиторию.
- **Варианты:**
  - (А) Продолжать работать с крупными сайтами, вкладываться в stealth Playwright и обход WAF
  - (Б) Сменить нишу валидации на средние компании без WAF
- **Решение:** вариант Б. Ниша — онлайн-образование (foxford, interneturok, webium, tutoronline, egeturbo, el-ed). Первый эталонный сайт — `el-ed.ru`, как наиболее полно размеченный в golden set (35 из 37 пунктов). Маппинг golden set → CHECK_ID зафиксирован в `GOLDEN_SET_MAPPING.md`. Эталонная таблица — `tests/fixtures/golden_set_v1.ods`.
- **Причина:** онлайн-образование — реальная целевая ЦА (активно собирают ПД, нет юротделов, нет дорогих WAF). el-ed.ru технически доступен для обычного SiteScanner, что позволяет валидировать сканер на реальном содержании, а не бороться с транспортным слоем.
- **Компромиссы:** валидация на одном сайте не покроет все edge cases; расширение до полного golden set (6 сайтов) — следующий этап после первого прогона.
- **Связанные файлы:** `GOLDEN_SET_MAPPING.md`, `tests/fixtures/golden_set_v1.ods`

### DEC-004: Стратегия извлечения текста из PDF-политик — каскад с Yandex Vision OCR
- **Дата:** 2026-04-10
- **Контекст:** Bug B на el-ed.ru — PDF без ToUnicode mapping, ни pdfplumber, ни pypdf, ни pdfminer.six не могут извлечь Unicode-кириллицу. Этот паттерн встречается в нише онлайн-образования системно. Без OCR значимая часть целевой ниши получает `pp.found=false` или `manual_review_needed`.
- **Варианты:**
  - (А) Локальный Tesseract — требует системных зависимостей poppler+tesseract, проблемы с кириллицей в путях на Windows, среднее качество на русском
  - **(Б) Облачный OCR (Yandex Vision)** ✓ — нулевые системные зависимости, отличное качество на русских офисных документах, копеечная цена
  - (В) Не делать OCR, всегда возвращать `manual_review_needed` для нечитаемых PDF — компромисс, не соответствует репутационной позиции проекта
- **Решение:** вариант Б — Yandex Vision OCR как primary OCR-слой. Двухуровневая схема: Yandex Vision → если сервис недоступен или вернул ошибку → деградация в `manual_review_needed` (вариант В как fallback на случай сбоя облака).
- **Причина:** Yandex Vision не требует установки на dev-машинах команды, качество OCR на русских политиках конфиденциальности высокое, цена для нашего объёма смешная (~13 коп/страница), архитектурно совместим с принципом проекта про соответствие 152-ФЗ (Yandex Cloud имеет обязательства по защите ПД по российскому законодательству).
- **Компромиссы:** зависимость от внешнего сервиса (закрывается fallback'ом в `manual_review_needed`), необходимость API-ключа в `.env`, минимальная стоимость в продакшне.
- **Связанные файлы:** `src/scanner/pdf_extractors.py`, `ScanMetadata`, `.env.example`
- **Связанные сессии:** 1.1 (архитектура и заглушка) → 1.2 (реализация)

### DEC-005: Принцип «необходимо и достаточно» в пользовательском интерфейсе
- **Дата:** 2026-04-10
- **Контекст:** при обсуждении того, показывать ли пользователю в отчёте пометку «текст политики извлечён через OCR», возник более общий вопрос о том, какую техническую информацию вообще выводить.
- **Решение:** пользователю в отчёте выводится только результат проверки и понятные рекомендации. Технические детали — какой краулер сработал (httpx vs Playwright), какой PDF-экстрактор сработал (pdfplumber vs Yandex Vision), какая модель LLM ответила, через какие fallback-уровни прошёл запрос — НЕ показываются пользователю. Они остаются во внутренних логах и метаданных для диагностики.
- **Причина:** подавляющее большинство пользователей — не технические специалисты. Техническая информация либо непонятна, либо вызывает лишние вопросы, либо отвлекает от сути отчёта (соответствие 152-ФЗ). Принцип «необходимо и достаточно».
- **Применение:** правило сквозное, применяется ко всем будущим решениям об интерфейсе и тексте отчётов.
- **Связанные файлы:** `src/scanner/pdf_extractors.py`, `ScanMetadata.extraction_method` (внутренний, не выводится в UI)
