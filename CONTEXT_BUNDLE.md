PROJECT PASSPORT — Compliance 152-ФЗ

**Последнее обновление:** 2026-04-06, первый валидационный прогон на el-ed.ru

Суть проекта
AI-сканер сайтов на соответствие требованиям 152-ФЗ «О персональных данных». Автоматически проверяет сайт, выявляет нарушения, рассчитывает штрафы по КоАП и генерирует пакет документов для устранения нарушений.

Целевая аудитория
Компании, которые собирают персональные данные на сайте и хотят проверить соответствие 152-ФЗ без дорогостоящего юридического аудита.

Ценностное предложение
За 1–2 минуты получить: score соответствия, список конкретных нарушений, расчёт потенциального штрафа и готовые документы для исправления ситуации.

Технический стек
| Компонент   | Технология                                                        |
| ----------- | ----------------------------------------------------------------- |
| Backend     | Python, FastAPI                                                   |
| Краулер     | httpx + BeautifulSoup (статика), Playwright (JS/трекеры)          |
| PDF         | pdfplumber (извлечение текста из PDF-политик)                     |
| LLM         | OpenRouter API (google/gemini-2.5-pro, контекст 1M токенов)       |
| База данных | SQLite (compliance.db)                                            |
| Web UI      | HTML, CSS, JS, marked.js                                          |
| Планировщик | APScheduler (реализован в модуле Competitor Intelligence Monitor) |
| Авторизация | Bearer token middleware                                           |
| Экспорт     | DOCX, PDF (кириллица — Times New Roman)                           |

Архитектура
URL сайта
    ↓
Краулер (crawler.py / playwright_crawler.py)
    — статический обход: httpx + BeautifulSoup
    — JS-обход: Playwright (трекеры, динамические баннеры)
    — PDF-политики: pdfplumber
    — ищет формы, скрипты, cookie-баннеры, политику конфиденциальности
    — собирает всех кандидатов политики, выбирает лучшую (_select_best_policy)
    ↓
Анализатор (analyzer.py) — проверяет 35+ пунктов чеклиста,
    считает score и штрафы по КоАП
    ↓
LLM (OpenRouter / gemini-2.5-pro) — формирует текстовый анализ с рекомендациями
    ↓
Генератор (generator.py) — создаёт пакет документов из 12 шаблонов
    (fallback: шаблонный документ с дисклеймером, если LLM недоступен)
    ↓
Web UI (/check) — отображает результат пользователю

# CLAUDE_CODE_RULES.md — Правила работы с Claude Code

> Этот документ — общая инструкция для начала каждой сессии с Claude Code.
> Скопируй его в корень любого проекта и отправь Claude Code в начале работы.

---

## 1. Главный принцип: НЕ НАВРЕДИ

- **Перед изменением любого файла** — убедись, что понимаешь, что он делает сейчас.
- **Не трогай код, который не относится к текущей задаче.** Даже если видишь «улучшение» — запиши его в отчёте, но не применяй без согласования.
- **Не удаляй и не переименовывай** существующие функции, классы, переменные, если это не является явной частью задачи.
- **Перед редактированием файла** — если нет уверенности, что сможешь откатить изменения через git, создай резервную копию: `cp файл.py файл.py.bak`. Лучше перестраховаться.

---

## 2. Тесты — твоя страховка

- **Перед началом работы** — прогони все существующие тесты (`pytest`). Убедись, что они проходят. Сообщи результат.
- **После каждого изменения** — прогони тесты снова. Если что-то сломалось — исправь, прежде чем двигаться дальше.
- **Не изменяй существующие тесты**, если это не часть задачи. Если тест «мешает» — это сигнал, что ты ломаешь рабочий код.
- **Не изменяй тестовые фикстуры и conftest.py**, чтобы подогнать тесты под изменённый код. Это та же ошибка, что и менять тесты, только замаскированная.
- **Новый код = новые тесты.** Каждая новая функция или метод должны сопровождаться тестами.

---

## 3. Работай пошагово

- **Одна задача — один шаг.** Не делай три вещи одновременно.
- **Лимит объёма:** если одно изменение затрагивает больше ~100 строк — остановись и согласуй план. Большие правки невозможно проверить глазами.
- **После каждого шага** — сообщи результат и жди подтверждения перед следующим шагом (формат отчёта — в разделе 9).

---

## 4. Не отклоняйся от задачи

- **Делай ровно то, что указано в задании.** Не больше и не меньше.
- **Не проводи «попутный рефакторинг»** — не переименовывай переменные, не меняй форматирование, не «улучшай» импорты в файлах, которые не относятся к задаче.
- **Не делай косметических правок:** не меняй кавычки (одинарные ↔ двойные), не пересортировывай импорты, не добавляй/убирай пустые строки, не трогай стиль кода в файлах, которые не являются целью задачи. `git diff` должен показывать только содержательные изменения.
- Если видишь проблему вне текущей задачи — **запиши её** в отчёте в пункте «Замечания вне текущей задачи», но не исправляй.

---

## 5. Обратная совместимость

- **Сигнатуры публичных функций и методов** (параметры, возвращаемые типы) — не менять без явного указания.
- **Модели данных** (dataclass, Pydantic) — новые поля добавлять только с дефолтным значением, чтобы существующий код не падал.
- **Импорты** — если выносишь функцию в другой модуль, оставь re-export в старом месте, чтобы не ломать вызовы из других файлов.
- **API-эндпоинты** — не менять URL, параметры, формат ответа без явного указания.

## 13. Git-гигиена: ритуалы сессии

**Главный принцип:** источник правды о состоянии проекта — это `origin` на GitHub, 
а не локальный диск и не PROJECT_PASSPORT.md. Если работы нет в `git log origin/...` — 
её формально нет, даже если код работает и тесты зелёные. Этот раздел — процедуры, 
которые защищают работу от потери между сессиями.

### Ритуал начала сессии

В первом сообщении каждой новой сессии, **до** любых правок кода, выполни и покажи:
```
git branch --show-current
git status
git log -3 --oneline
```

Дай человеческое резюме в 2–3 строках: «мы в ветке X, последний коммит — Y от такой-то 
даты, дерево чистое/грязное, синхронизировано с origin / есть N локальных коммитов 
без пуша».

**«Чистый git status» означает:** нет `modified`-файлов и нет код-релевантных 
`untracked`-файлов. Эти untracked допустимы и не требуют действий:
- `docs/sessions/SESSION_*.md`, `docs/tasks/TASK_*.md` — локальные заметки рабочей сессии
- `.claude/`, `server.log`, `*.pyc`, `__pycache__/` — локальные артефакты
- всё, что уже перечислено в `.gitignore`

**Если git status не чистый** (есть `modified` или код-релевантные `untracked`): 
**остановись и не начинай новую задачу**, пока не разобрался, откуда взялись изменения. 
Три возможных пути, выбираются вместе с пользователем:
1. Это работа предыдущих сессий, которая забыла закоммититься → закоммитить 
   (с разбиением на логические единицы, если правок много)
2. Это незавершённый эксперимент, который нужно сохранить → `git stash save "описание"` 
   или перенести в ветку `wip/...`
3. Это артефакты для удаления → точечно, только после явного подтверждения пользователя

**Не начинай новую работу поверх неразобранного состояния.** Это правило без исключений.

**Если есть локальные коммиты без пуша** (`Your branch is ahead of origin/... by N commits`): 
предложи `git push` как первое действие сессии.

### Работа по ходу сессии

- **После каждого завершённого пункта плана** (не «правка файла», а именно пункт из задачи, 
  которую дал пользователь) — предложи коммит с понятным сообщением. Не копи изменения 
  до конца сессии.
- **Перед каждым коммитом — прогон тестов.** Красные тесты не коммитятся никогда, ни под 
  каким соусом «потом починим». Если тесты упали — остановись, покажи ошибку, разбирайся.
- **Про согласование коммитов:** Claude Code **предлагает** коммит и сообщение, пользователь 
  одобряет (достаточно «да» или «коммить»), дальше коммит делается автоматически. 
  Это не противоречит принципу «не коммитить без согласования» — согласование есть, 
  просто облегчённое.
- **Размер коммита:** одна смысловая единица. Если одним шагом затронуто больше ~10 файлов 
  или разные подсистемы — предложи разбить на несколько коммитов.

### Ритуал конца сессии

Перед завершением работы выполни и покажи:
```
git status
git log -3 --oneline
```

**Идеальный финал:** `git status` пустой (с точностью до разрешённых untracked), последние 
коммиты сессии видны в `git log`, ветка `up to date with origin/...` — то есть всё запушено.

**Если работа не закончена и коммитить нельзя** (незавершённый эксперимент, половина задачи): 
явно скажи об этом пользователю и предложи `git stash save "описание"` или создание ветки 
`wip/имя-задачи`. Не оставляй пользователя в неявно грязном состоянии — он должен сознательно 
решить, что делать с незавершённой работой.

**Локальные коммиты без пуша — не финал.** Перед концом сессии обязательный `git push`.

### Правило паспорта: «готово» = в origin

Статус `✅ Готово` в `PROJECT_PASSPORT.md` можно ставить для задачи **только при 
выполнении обоих условий одновременно:**
1. Соответствующий коммит виден в `git log`
2. Коммит запушен в `origin` (проверить: `git status` говорит `up to date with origin/...`)

«Тесты зелёные + код работает» — это ещё не «готово». Это «готово фактически, 
не готово формально». Разрыв между этими двумя состояниями — главный источник потерь 
в истории проекта.

Для существующих записей в паспорте правило не применяется ретроактивно — прошлое 
не переписываем. Только для новых записей.

### CASE-001: wildberries.ru — сайт полностью недоступен из-за анти-бота
- ★ **Дата:** 2026-04-04
- ★ **URL:** https://wildberries.ru
- ★ **Категория:** policy-not-found
- ★ **Что произошло:** сканер вернул 0 страниц, политика не найдена, ошибка «illegal status line» с HTTP 498
- ★ **Что ожидалось:** политика существует по адресу `/services/privacypolicy/general-terms-of-use/` (проверено curl)
- **Где на сайте:** отдельная страница, путь `/services/privacypolicy/general-terms-of-use/`
- **Краулер:** SiteScanner (httpx)
- **Причина:** Wildberries возвращает нестандартный HTTP 498 + JS-challenge для всех не-браузерных запросов; httpx получает тело HTML как статусную строку и бросает исключение `illegal status line`
- **PlaywrightCrawler:** частично лучше — `pages_scanned=1`, `errors=[]` (vs 0 страниц и ошибка у httpx). Playwright прошёл HTTP 498, загрузил JS-challenge страницу, но bot-detection WB сработал и до реального сайта добраться не удалось. `pp.found=False`, 0 форм, 2 скрипта (скрипты JS-challenge)
- **Статус:** открыт — Playwright не обходит WB anti-bot полностью; нужен реалистичный UA + возможно stealth-режим

### CASE-002: tinkoff.ru — политика найдена, но текст не извлечён (JS-рендеринг)
- ★ **Дата:** 2026-04-04
- ★ **URL:** https://tinkoff.ru
- ★ **Категория:** score-wrong
- ★ **Что произошло:** `privacy_policy.found=True`, URL верный (`/privacy/`), но текст политики — HTML-заглушка «Включите JavaScript»; все поля анализа содержания (`has_data_categories`, `has_purposes`, `has_subject_rights`, `is_russian`) = False
- ★ **Что ожидалось:** текст политики извлечён, поля анализа заполнены корректно
- **Где на сайте:** https://tinkoff.ru/privacy/ — страница требует JS для отображения контента
- **Краулер:** SiteScanner (httpx)
- **Причина:** страница политики рендерится через JavaScript; httpx получает пустой HTML без контента; `is_russian=False` → score за содержание политики = 0 несмотря на наличие ссылки
- **PlaywrightCrawler (до смены UA):** тот же текст-заглушка (`text_len=6722`, `is_russian=False`). Детектирует бота по UA `Compliance152Bot/0.1`.
- **PlaywrightCrawler (после смены UA на Chrome/120):** результат идентичен — `text_len=6722`, `is_russian=False`. Смена UA не помогла. Причина: Tinkoff детектирует headless-режим через `navigator.webdriver=true` — это стандартное свойство Playwright, не зависящее от UA. Нужен stealth-режим: запуск с `--disable-blink-features=AutomationControlled` + патч `navigator.webdriver`.
- **Статус:** открыт — требуется stealth-режим Playwright, не просто смена UA

### CASE-003: ozon.ru — HTTP 403, страница засчитана как просканированная без ошибки
- ★ **Дата:** 2026-04-04
- ★ **URL:** https://ozon.ru
- ★ **Категория:** policy-not-found
- ★ **Что произошло:** сканер вернул `pages_scanned=1`, `errors=[]`, политика не найдена; фактически сайт вернул HTTP 403 с JS-challenge страницей («An...») для всех путей включая `/t/privacy-notice`, `/legal/privacy`, `/privacy`
- ★ **Что ожидалось:** ошибка доступа должна быть зафиксирована; `pages_scanned` не должен засчитывать 403-страницы как успешно просканированные
- **Где на сайте:** https://ozon.ru — 403 Forbidden на всём домене при запросах без browser fingerprint
- **Краулер:** SiteScanner (httpx)
- **Причина:** Ozon использует bot-protection (403 + Cloudflare/собственный WAF); httpx не отдаёт нужный fingerprint браузера; 403-ответ засчитывается сканером как «страница просканирована» без записи в `errors[]`
- **PlaywrightCrawler:** прошёл 403-барьер (`pages_scanned=2`, `errors=[]`), но обнаружил **ложноположительную политику**: `pp.found=True`, `pp.url=https://www.ozon.ru/complaint/support/?incident_id=fab_chlg_...` — это страница bot-challenge, не политика конфиденциальности. `text_len=211`, `is_russian=False`. Playwright справился с HTTP-барьером, но создал новую проблему: ложный `pp.found=True` хуже, чем честный `pp.found=False`
- **Отдельный баг (валидация политики):** `is_privacy_policy_page()` или механизм выбора кандидата принял challenge-URL за политику. Нужна валидация: найденная «политика» должна содержать признаки реального документа (минимальная длина текста, наличие ключевых слов 152-ФЗ, `is_russian=True`), иначе `pp.found` должен оставаться `False`.
- **Статус:** открыт — три независимых бага: (1) HTTP 403 не в errors для SiteScanner; (2) ложный `pp.found=True` при детектировании challenge-страниц; (3) нет валидации качества найденного документа политики

### CASE-004: mos.ru — ConnectTimeout через системный прокси, пустое сообщение об ошибке
- ★ **Дата:** 2026-04-04
- ★ **URL:** https://mos.ru
- ★ **Категория:** crash
- ★ **Что произошло:** сканер вернул `pages_scanned=0`, `errors=['https://mos.ru: ']` (пустое сообщение); политика не найдена
- ★ **Что ожидалось:** либо успешное сканирование, либо ошибка с читаемым сообщением (`ConnectTimeout` или `ProxyTimeout`)
- **Где на сайте:** главная страница — до неё не добрались

- **Статус:** CASE условно закрыт для mos.ru (сетевая проблема окружения, не баг). Баги в сканере: (1) `str(ConnectTimeout)=''` — ошибка логируется без типа исключения (актуально); (2) httpx `trust_env=True` подхватывает системный прокси (актуально для всех окружений с прокси)

## Решения

### DEC-001: User-Agent — гибридный подход
- **Дата:** 2026-04-05
- **Контекст:** тестирование на 5 крупных сайтах показало, что `Compliance152Bot/0.1` блокируется на 4 из 5 (WB, Tinkoff, Ozon, DNS). PlaywrightCrawler запускает реальный Chromium — браузерный UA здесь честен технически.
- **Варианты:**
  - (А) Честный бот-UA везде — сильно ограничивает охват, большинство крупных сайтов недоступны
  - (Б) Браузерный UA везде (в т.ч. httpx) — в httpx это бесполезный обман, WAF всё равно детектирует по отсутствию JS и fingerprint
  - **(В) Гибрид: SiteScanner — честный UA, PlaywrightCrawler — браузерный UA** ✓
- **Решение:** вариант В. SiteScanner сохраняет `Compliance152Bot/0.1`. PlaywrightCrawler использует реальный браузерный User-Agent (это не обман — он и есть настоящий Chromium).
- **Компромиссы:** часть сайтов всё равно детектирует headless-браузеры по другим признакам (canvas fingerprint, отсутствие плагинов и т.д.). Полного обхода не гарантировано.
- **Примечание для отчётов:** режим сканирования (httpx vs Playwright) пользователю не показывать — он пришёл проверить соответствие 152-ФЗ, а не разбираться в технических деталях.
- **Связанные файлы:** `src/scanner/playwright_crawler.py` (строка с `user_agent=`)

---

### DEC-002: Автоматический fallback SiteScanner → PlaywrightCrawler
- **Дата:** 2026-04-05
- **Контекст:** тестирование показало, что SiteScanner возвращает «плохой» результат (0 страниц, 4xx, пустой текст политики) на крупных сайтах без явной ошибки для пользователя. Ручная кнопка «попробовать в браузерном режиме» не подходит — пользователь не поймёт зачем.
- **Варианты:**
  - (А) Ручная кнопка — требует от пользователя понимания разницы краулеров, большинство уйдут с пустым результатом
  - **(Б) Автоматический fallback** ✓
- **Решение:** вариант Б. Если SiteScanner вернул «плохой» результат — автоматически перезапускать через PlaywrightCrawler. Критерии плохого результата: `pages_scanned=0`, или наличие 4xx в errors, или `privacy_policy.found=False` при `pages_scanned≥1` и `text_len<100`.

### Крупные российские сайты блокируют SiteScanner (httpx) через WAF

**Контекст:** тестирование сканера на реальных сайтах (2026-04-04).

- DNS Shop: HTTP 401 (Qrator WAF) → то же молчаливое поведение (CASE-005)

### AsyncClient mock для SiteScanner smoke-тестов

**Контекст:** написание smoke-тестов для `SiteScanner.scan()` без реальных HTTP-запросов.

**Проблема:** `scan()` создаёт `httpx.AsyncClient` через `async with` внутри метода — патчить клиент напрямую нельзя, нужно патчить точку создания. Кроме того, метод делает два типа запросов: `HEAD` (для `_check_ssl`) и `GET` (для страниц).

**Решение:** патчить `src.scanner.crawler.httpx.AsyncClient`, возвращая `AsyncMock`, у которого `__aenter__` возвращает сам себя, а `head` и `get` — разные заготовленные ответы.

```python
mock_client = AsyncMock()
mock_client.head = AsyncMock(return_value=ssl_resp)
mock_client.get = AsyncMock(return_value=main_resp)
mock_client.__aenter__ = AsyncMock(return_value=mock_client)
mock_client.__aexit__ = AsyncMock(return_value=False)

with patch("src.scanner.crawler.httpx.AsyncClient", return_value=mock_client):
    result = await scanner.scan("https://example.com")
```

**Где применено:** `tests/test_crawler.py` — все тесты групп A, B, C.

**Когда пригодится снова:** любой тест, покрывающий метод, который создаёт `httpx.AsyncClient` внутри себя через `async with`. Паттерн — патчить в модуле-потребителе, не в `httpx`.

### Мок Playwright page.on('request') для тестирования перехвата запросов

**Контекст:** тестирование `PlaywrightCrawler._crawl()` без реального браузера.

**Проблема:** `page.on('request', handler)` регистрирует sync callback, который Playwright вызывает при каждом сетевом запросе во время `goto()`. Стандартный `AsyncMock` не умеет «стрелять» событиями — handler не вызывается, `request_domains` остаётся пустым.

**Решение:** `_MockPage` — минимальный класс-заглушка:
- `on(event, handler)` — сохраняет handler в dict
- `goto(url, ...)` — вызывает сохранённый handler вручную для каждого URL из fixture

```python
class _MockPage:
    def on(self, event: str, handler) -> None:
        self._handlers[event] = handler

    async def goto(self, url: str, **kwargs):
        for req_url in self._request_urls:
            req = MagicMock()
            req.url = req_url
            if "request" in self._handlers:
                self._handlers["request"](req)
        return MagicMock(status=200)
```

**Где применено:** `tests/test_playwright_crawler.py`.

**Когда пригодится снова:** любой тест Playwright-кода с `page.on()` — request, response, dialog, navigation events. Общий паттерн: `on()` сохраняет, триггер-метод (`goto()`/`click()`) вызывает.

### PlaywrightCrawler + page.content() = JS-формы без изменений в детекторах

**Контекст:** задача 6.4 — добавление поддержки JS-форм (Tilda, Bitrix) и JS-баннеров (OneTrust, CookieYes).

**Проблема:** казалось, что `extract_forms()` не найдёт формы Tilda/Bitrix, потому что они монтируются через JavaScript и отсутствуют в исходном HTML.

**Решение:** PlaywrightCrawler уже вызывает `html = await page.content()` после `await asyncio.sleep(2)` — это **рендеренный DOM** после выполнения JS. BeautifulSoup получает полный DOM, включая все JS-монтированные элементы. Стандартный `soup.find_all("form")` в `extract_forms()` находит Tilda (`<div class="t-form"><form ...>`) и Bitrix (`<div class="b24-form"><form ...>`) без каких-либо изменений в детекторах.

**Где применено:** `src/scanner/playwright_crawler.py:154`, `src/scanner/detectors.py` — `extract_forms()`.

**Когда пригодится снова:** при любом обсуждении «нужен ли отдельный JS-form detector» — нет, не нужен. Вся магия в `page.content()`, который возвращает пост-JS DOM. Отдельный детектор для JS-форм был бы избыточным усложнением.

## Замечания для обсуждения

### A. Нет прямого маппинга для пункта 8 «Выбор категорий cookie»

В чеклисте сканера есть `COOKIE_003` — «Аналитические/маркетинговые cookie выключены по умолчанию». Это смежный, но не идентичный пункт. «Выбор категорий» подразумевает наличие UI с чекбоксами «аналитика / маркетинг / функциональные» в самом баннере. «Выключены по умолчанию» — что при открытии баннера эти категории уже в состоянии off. У el-ed.ru нет ни того, ни другого (обычный баннер с «Понятно»), так что для конкретного сайта FAIL = FAIL — расхождения не будет. Но в других сайтах из golden set возможна ситуация «категории есть, но включены по умолчанию» — тогда по эталону ✅, а по сканеру FAIL. Это нужно держать в голове.

### B. Пункт 34 — кандидат на новый CHECK_ID

Ты правильно зафиксировала в эталоне противоречие el-ed: «в политике написано, что трансграничной передачи нет, но фактически подключены GA, Google Fonts, GTM». Это именно тот тип нарушения, за который пользователь будет платить сканеру. В текущем чеклисте ближайший чек — `TRACKER_001` (трекер без упоминания в политике), но он про другое: «сам факт наличия трекера не упомянут». А здесь — «упомянут отдельный аспект (трансграничная передача), который явно противоречит наличию трекеров».

**Предложение:** добавить в чеклист новый пункт, например `POLICY_018` — «Непротиворечивость раздела о трансграничной передаче с реально подключёнными сервисами». Severity — `high` или `critical`. Method — `llm_check` (сравнение текста политики с `scan_result.trackers`). Это будет отдельное решение в `DECISIONS.md`, и его имеет смысл делать после первого прогона — когда мы увидим вживую, как сканер сейчас реагирует на el-ed.

### C. Сканер найдёт больше, чем есть в эталоне

В чеклисте есть категория `consent` (CONSENT_001–005), которой в golden set нет как отдельных строк (они «спрятаны» внутри пункта 1). CONSENT_002 «Согласие отделено от оферты», CONSENT_003 «Обязательные реквизиты», CONSENT_004 «Возможность отзыва», CONSENT_005 «Раздельные согласия для разных целей» — сканер их проверит, и в отчёте появятся либо PASS, либо FAIL, которых в golden set просто нет. **Это не расхождение, это ожидаемо.** Сканер шире, чем эталон — и это хорошо.

### D. Инвертированная семантика TECH_*

Важно при сравнении: в golden set по пунктам 28–32 `✅ = сервис обнаружен`. В чеклисте сканера TECH_002…TECH_006 сформулированы как **«Отсутствие X»**, то есть `PASS = отсутствует, всё хорошо`, `FAIL = обнаружен, нарушение`. Это противоположности. При автоматическом сравнении нужно инвертировать: `golden[28]=✅` → `expected: FAIL по TECH_002`.

### E. REG_001, REG_002, REG_003 — исключаем из первого прогона

Все три пункта категории `regulatory` помечены в чеклисте как `method: manual`. Сканер их не проверяет. В эталоне REG_001 и REG_002 не заполнены (ты сама написала, что это сейчас некритично), REG_003 заполнен (✅). При первом прогоне эти три строки просто пропускаем — ни PASS, ни FAIL, а статус «не покрывается сканером». В будущем — REG_003 можно автоматизировать через whois/IP-геолокацию, это отдельная небольшая задача.

## Почему именно такой порядок

Порядок не случайный. Между сессиями есть зависимости, и нарушение порядка

## Сессия 2 — LLM-инфраструктура: модель и fallback-цепочка

**Приоритет:** высокий. Связан с CASE-006.
**Ожидаемая длина:** 1 сессия.
**Предпосылка:** сессия 1 закрыта, pp.found=true на el-ed.ru работает.

### Контекст

В сессии 2026-04-06 обнаружено (CASE-006), что LLM-клиент использует
`openrouter/free` вместо `google/gemini-2.5-pro`, указанного в конфиге.
Свободный роутинг OpenRouter отдаёт запрос произвольной модели — ответ
получаем, но качество «как повезёт». Это было осознанным выбором на этапе
прототипа (лучше какой-то ответ, чем падение), но сейчас проект вырос,
и нужен нормальный баланс надёжности и качества.

Решение: передавать не одну модель, а упорядоченную **цепочку** моделей
через параметр `models` в OpenRouter API. Если первая недоступна —
автоматический переход ко второй, к третьей. Это даёт надёжность без
жертвы качеством.

### Шаги

1. **Диагностика CASE-006.** Найти в коде, где именно формируется параметр
   `model` для вызова OpenRouter. Понять, почему он становится `openrouter/free`
   несмотря на `OPENROUTER_MODEL=google/gemini-2.5-pro` в `.env`. Возможные
   причины: (а) код явно подставляет fallback при ошибке загрузки конфига,
   (б) ключ OpenRouter имеет ограничения tier'а, (в) где-то осталась
   hardcoded-строка от прототипа.

2. **Обновление актуального каталога.** Перед выбором конкретных моделей —
   свериться с актуальной документацией OpenRouter и их каталогом моделей.
   Каталог меняется, и версии, актуальные на момент создания этого плана,
   к моменту выполнения могут устареть. Это лучше делать через web_search
   перед началом работы.

3. **Формирование цепочки.** По итогам шага 2 собрать упорядоченный список
   моделей. Ориентировочная структура:
   - **Primary:** Gemini 2.5 Pro (контекст 1M — важно для длинных политик)
   - **Fallback 1:** Gemini 2.5 Flash (та же семья, дешевле, быстрее)
   - **Fallback 2:** Claude Sonnet актуальной версии или DeepSeek
     (на случай проблем у Google на стороне провайдера)

   Конкретные model ID сверять с каталогом OpenRouter в момент внедрения.

4. **Правка кода вызова LLM.** Передавать `models: [...]` вместо `model: ...`
   в тело запроса. Если текущий клиент — обёртка поверх httpx, добавить
   поддержку массива. Если используется openai SDK в режиме OpenRouter —
   проверить, поддерживает ли он `extra_body={"models": [...]}`.

5. **Логирование реальной модели.** В логах вызова LLM обязательно фиксировать
   поле `model` из ответа OpenRouter — какая модель фактически ответила.
   Это критично для отладки и для возможного кэша (см. сессию 4).

6. **Обновление `.env.example` и `PROJECT_PASSPORT`.** Документировать новую
   схему конфига (какие переменные задают primary и fallback модели).

7. **Закрытие CASE-006.** После починки обновить статус кейса на «исправлено»
   с указанием сессии и реальной причины.

### Критерии успеха

- Тестовый вызов LLM через код проекта возвращает ответ от Gemini 2.5 Pro
  (или от следующего в цепочке, если Pro временно недоступен)
- В логах видна реальная модель, которая ответила
- pytest зелёный
- CASE-006 закрыт

### Чего не делать в этой сессии

- Не делать LLM-кэш (это сессия 4)
- Не разделять вызовы LLM по типу задачи (это тюнинг для более поздних сессий)
- Не трогать промпты — только инфраструктуру вызова

## Сессия 4 — LLM-кэш по text_hash

**Приоритет:** средний, но с высоким ROI.
**Ожидаемая длина:** 1 сессия.
**Предпосылка:** сессии 1–3 закрыты, модель работает корректно.

### Контекст

LLM-анализ — самая дорогая часть пайплайна. Политики конфиденциальности
меняются редко (раз в полгода или реже). При повторных сканированиях одного
и того же сайта мы сейчас каждый раз тратим деньги и время на LLM-вызов,
хотя текст политики не изменился.

У проекта уже есть вся инфраструктура для кэша: `ScanMetadata.text_hash`
считается при извлечении политики (провенанс документа), SQLite поднят.
Не хватает только таблицы кэша и обёртки вокруг вызова LLM.

Это особенно важно для Monitor/Updater из паспорта — он будет регулярно
пересканировать сайты, и без кэша превратит проект в бесконечный расход
на OpenRouter.

### Шаги

1. **Дизайн таблицы кэша.** Предложить структуру:
   ```
   llm_cache (
     text_hash TEXT,
     model TEXT,
     prompt_version TEXT,
     response_json TEXT,
     created_at TIMESTAMP,
     PRIMARY KEY (text_hash, model, prompt_version)
   )
   ```
   Ключевой момент: `prompt_version` обязателен. Если промпт изменился —
   кэш должен инвалидироваться автоматически, иначе получим старые ответы
   на новые вопросы.

2. **Миграция БД.** Добавить таблицу в `compliance.db`. Поскольку проект
   пока MVP и структура БД не зафиксирована жёсткими миграциями — проверить,
   как сейчас в проекте принято изменять схему (есть ли Alembic или
   голые SQL-миграции).

3. **Обёртка вокруг вызова LLM.** Функция вида `analyze_policy_cached(text, ...)`,
   которая:
   - Считает hash текста (или берёт готовый из `ScanMetadata`)
   - Смотрит в кэш по тройке `(hash, model, prompt_version)`
   - Если нашла — возвращает сохранённый ответ, логирует `cache_hit=True`
   - Если не нашла — вызывает LLM, сохраняет ответ в кэш, возвращает его

4. **Версионирование промптов.** Ввести константу `PROMPT_VERSION` в модуле
   с промптами. Любая правка промпта = инкремент версии. Старые кэши
   остаются в БД для истории, но не используются.

5. **Тесты на кэш.** Два сценария:
   - Первый вызов → cache miss → запись в БД → возврат ответа
   - Повторный вызов с тем же хэшем → cache hit → БД не пополняется,
     LLM не вызывается (проверить через mock)

6. **Метрики.** Добавить в лог скана количество cache hits и cache misses
   за прогон. Это поможет видеть эффект в реальных данных.

### Критерии успеха

- Повторный прогон `run_golden_scan.py` на том же сайте без изменений
  политики не делает ни одного вызова LLM (всё из кэша)
- Изменение `PROMPT_VERSION` автоматически инвалидирует кэш
- pytest зелёный

### Чего не делать в этой сессии

- Не кэшировать HTML/сам результат скана — только LLM-ответы. Скан должен
  оставаться свежим, потому что меняются трекеры, куки, формы.
- Не делать TTL на кэше. Инвалидация только по `text_hash` (политика
  изменилась → новый хэш → новый вызов). Временная инвалидация нам
  не нужна — политика либо та же, либо другая.

После того как el-ed.ru полностью «позеленеет» (сессии 1–4), открывается
backlog, который сейчас не в приоритете, но зафиксирован чтобы не потерялся.

### Расширение golden set на остальные сайты ниши

Досопоставить остальные сайты из колонки golden set (foxford, interneturok,
webium, tutoronline, egeturbo) с CHECK_ID. Прогнать сканер на каждом.
Построить матрицу совпадений. Это даст первое представление о точности
сканера в нише онлайн-образования в целом, а не только на одном сайте.

### Новые CHECK_ID по итогам первого прогона

Два конкретных кандидата уже есть:
- **POLICY_018** — непротиворечивость раздела о трансграничной передаче
  с реально подключёнными сервисами. Method: `llm_check`. Severity: high
  или critical. (Пункт 34 из GOLDEN_SET_MAPPING, замечание B.)
- **TECH_007** — Yandex SmartCaptcha. Аналог TECH_005 (Google reCAPTCHA)
  для российского рынка. Method: автоматически по заголовкам/скриптам.

Оба добавлять как отдельные решения в `DECISIONS.md` (номера присваиваются
по факту в момент записи).

### Разделение LLM-вызовов по типу задачи

После того как базовая цепочка моделей и кэш работают — можно подумать
о разделении:
- **Извлечение** (анализ политики на наличие обязательных разделов) —
  более дешёвая модель (Flash, DeepSeek)
- **Генерация** (документы-шаблоны для устранения нарушений) — более
  качественная модель (Pro, Sonnet)

Это может сократить расходы ещё вдвое. Не трогать до тех пор, пока
базовая инфраструктура не стабильна.

### Задачи D, E, F из PROJECT_PASSPORT (краулер)

- **(D)** Корректный учёт 4xx в `errors[]`, не засчитывать в `pages_scanned`
- **(E)** Автоматический fallback SiteScanner → PlaywrightCrawler (DEC-002)
- **(F)** Stealth-режим Playwright для сайтов с bot fingerprinting (CASE-002)

Все три — про крупные сайты с WAF. Имеет смысл делать после того, как
валидация на el-ed и остальных сайтах образовательной ниши завершена.
До этого момента они — шум, отвлекающий от главной задачи.

### Прочее из PROJECT_PASSPORT

- Синхронизация `_extract_privacy_policy` между краулерами
  (truncation 20k vs 100k, отсутствие text_hash в Playwright-версии)

- Полировка UI (мобильная адаптация, цвета риска, центровка score-круга)

## Как использовать этот файл

### В начале каждой новой сессии

1. Открыть `NEXT_SESSIONS_PLAN.md` и PROJECT_PASSPORT.md
2. Определить, какая сессия текущая (по порядку выполненных)
3. Прикрепить оба файла в начало разговора с Claude (для обсуждения)
4. Отдельно отправить Claude Code команду «прочитай
   `NEXT_SESSIONS_PLAN.md`, сессия N — это твоя задача» —
   чтобы у него сразу был контекст цели, критериев успеха
   и явного списка «чего не делать»

### После каждой завершённой сессии

- Обновить `PROJECT_PASSPORT.md` по §10 правил (отдельным коммитом)
- Отметить в этом файле (`NEXT_SESSIONS_PLAN.md`) статус сессии:
  `✅ Выполнено <дата>` рядом с заголовком
- Если по итогам возникли новые задачи — добавить их в backlog (сессия 5+)
- Если план изменился по итогам — переписать затронутую сессию целиком,
  не боясь ломать старую редакцию (история в Git)

### Когда план устарел

Этот файл — живой документ, а не высеченный в камне. Если по итогам
сессии 1 (pp.found=false) выяснится, что причина была совсем не в регрессии
валидации, а например в кодировке или редиректах — план сессии 2 может
остаться актуальным, а план сессии 3 нужно будет пересмотреть. Это нормально.

Главная ценность файла — не «идеальное предвидение», а **явная фиксация
того порядка, который казался правильным на момент создания**, чтобы в
начале каждой сессии не приходилось заново реконструировать «почему мы
делаем именно это именно сейчас».
