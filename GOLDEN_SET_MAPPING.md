# GOLDEN_SET_MAPPING.md — сопоставление эталона с чеклистом сканера

> **Зачем этот файл:** связать 37 пунктов ручной проверки из `golden_set.ods`
> с `CHECK_ID` сканера (`knowledge_base/checklists/website_checklist.json`, 47 пунктов),
> чтобы иметь возможность автоматически сравнивать результат сканера с эталоном.
>
> **Версия:** v0.1, 2026-04-05. Основа — колонка el-ed.ru.

---

## Условные обозначения

- **Авто** — сканер проверяет этот пункт автоматически
- **Частично** — сканер проверяет похожий, но не идентичный аспект
- **Руками** — сейчас только ручная проверка (`method: manual` в чеклисте)
- **Инв.** — инвертированная семантика: golden set «обнаружен = ✅», чеклист «отсутствует = ✅»

## Замечания для обсуждения

### A. Нет прямого маппинга для пункта 8 «Выбор категорий cookie»

В чеклисте сканера есть `COOKIE_003` — «Аналитические/маркетинговые cookie выключены по умолчанию». Это смежный, но не идентичный пункт. «Выбор категорий» подразумевает наличие UI с чекбоксами «аналитика / маркетинг / функциональные» в самом баннере. «Выключены по умолчанию» — что при открытии баннера эти категории уже в состоянии off. У el-ed.ru нет ни того, ни другого (обычный баннер с «Понятно»), так что для конкретного сайта FAIL = FAIL — расхождения не будет. Но в других сайтах из golden set возможна ситуация «категории есть, но включены по умолчанию» — тогда по эталону ✅, а по сканеру FAIL. Это нужно держать в голове.

### B. Пункт 34 — кандидат на новый CHECK_ID

Ты правильно зафиксировала в эталоне противоречие el-ed: «в политике написано, что трансграничной передачи нет, но фактически подключены GA, Google Fonts, GTM». Это именно тот тип нарушения, за который пользователь будет платить сканеру. В текущем чеклисте ближайший чек — `TRACKER_001` (трекер без упоминания в политике), но он про другое: «сам факт наличия трекера не упомянут». А здесь — «упомянут отдельный аспект (трансграничная передача), который явно противоречит наличию трекеров».

**Предложение:** добавить в чеклист новый пункт, например `POLICY_018` — «Непротиворечивость раздела о трансграничной передаче с реально подключёнными сервисами». Severity — `high` или `critical`. Method — `llm_check` (сравнение текста политики с `scan_result.trackers`). Это будет отдельное решение в `DECISIONS.md`, и его имеет смысл делать после первого прогона — когда мы увидим вживую, как сканер сейчас реагирует на el-ed.

### C. Сканер найдёт больше, чем есть в эталоне

В чеклисте есть категория `consent` (CONSENT_001–005), которой в golden set нет как отдельных строк (они «спрятаны» внутри пункта 1). CONSENT_002 «Согласие отделено от оферты», CONSENT_003 «Обязательные реквизиты», CONSENT_004 «Возможность отзыва», CONSENT_005 «Раздельные согласия для разных целей» — сканер их проверит, и в отчёте появятся либо PASS, либо FAIL, которых в golden set просто нет. **Это не расхождение, это ожидаемо.** Сканер шире, чем эталон — и это хорошо.

### D. Инвертированная семантика TECH_*

Важно при сравнении: в golden set по пунктам 28–32 `✅ = сервис обнаружен`. В чеклисте сканера TECH_002…TECH_006 сформулированы как **«Отсутствие X»**, то есть `PASS = отсутствует, всё хорошо`, `FAIL = обнаружен, нарушение`. Это противоположности. При автоматическом сравнении нужно инвертировать: `golden[28]=✅` → `expected: FAIL по TECH_002`.

### E. REG_001, REG_002, REG_003 — исключаем из первого прогона

Все три пункта категории `regulatory` помечены в чеклисте как `method: manual`. Сканер их не проверяет. В эталоне REG_001 и REG_002 не заполнены (ты сама написала, что это сейчас некритично), REG_003 заполнен (✅). При первом прогоне эти три строки просто пропускаем — ни PASS, ни FAIL, а статус «не покрывается сканером». В будущем — REG_003 можно автоматизировать через whois/IP-геолокацию, это отдельная небольшая задача.

## Ожидаемый результат первого прогона на el-ed.ru

Из 37 пунктов golden set:
- **34 пункта** сканер должен проверить автоматически
- **3 пункта** (REG_001, REG_002, REG_003) — вне зоны сканера

Из 34 автоматизируемых, по эталону:
- **PASS (всё ок):** 22 пункта
- **FAIL (нарушение):** 12 пунктов
  - FORM_002 (чекбокс предотмечен)
  - FORM_006 (нет отдельного чекбокса маркетинга)
  - COOKIE_002 (нет кнопки «Отклонить»)
  - COOKIE_003 (нет выбора категорий cookie)
  - COOKIE_005 (аналитика грузится до согласия)
  - POLICY_005 (не указан контакт ответственного)
  - TECH_002 (обнаружен Google Fonts)
  - TECH_003 (обнаружен Google Analytics)
  - TECH_005 (обнаружена reCAPTCHA)
  - TECH_006 (обнаружен GTM)
  - TRACKER_001 (трекеры частично упомянуты в политике)
  - *неявно:* POLICY_018 — противоречие по трансграничке (если будет добавлен)

**Score по эталону (грубая оценка):** 22/34 ≈ 65%. Это «средний» сайт по нише — не катастрофа, но явные проблемы с cookie-баннером, чекбоксами форм и необходимостью упомянуть трекеры в политике.

Если сканер на первом прогоне покажет score в диапазоне 60–70% — мы в зелёной зоне. Если сильно ниже — значит, сканер «находит» проблемы, которых по эталону нет (ложные срабатывания). Если сильно выше — значит, не замечает реальных проблем (пропуски). И то, и другое — материал для следующей сессии.
