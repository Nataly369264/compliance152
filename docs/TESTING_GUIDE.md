# Руководство по тестированию сканера Compliance 152

> Для коллег, которые проводят тестирование на реальных сайтах.  
> Цель тестирования — понять, насколько сканер заслуживает доверия: находит ли он реальные нарушения, не поднимает ли ложную тревогу, корректно ли работает в нештатных ситуациях. Баги — важная часть этой картины, но не единственная. Если всё работает правильно — это тоже ценный результат, а не провал тестирования.

---

## Зачем вообще тестировать и как это работает

Сканер — это программа, которая смотрит на сайт и говорит: «соответствует» или «не соответствует» по каждому из 42+ пунктов чеклиста 152-ФЗ. Она может ошибаться в двух направлениях:

- **Пропустить нарушение** — сказать «всё хорошо», хотя реально нарушение есть. Это ложное «не нашёл» (false negative). Для сканера на 152-ФЗ это хуже: клиент думает, что чист, а на самом деле нет.
- **Поднять ложную тревогу** — сказать «нарушение», хотя всё в порядке. Это ложная тревога (false positive). Клиент напрасно пугается и теряет доверие к инструменту.

Твоя задача при тестировании — зафиксировать оба типа расхождений. Без этого мы не знаем, в каком направлении улучшать сканер.

---

## Часть 1. Подготовка к тестовому сеансу

### Что нужно иметь перед началом

1. **Запущенный сервер и браузер — открой http://127.0.0.1:9000/check. Именно так выглядит сканер для конечного пользователя, именно в этом режиме и тестируем. Серверные логи при тестировании смотреть не нужно — это задача разработчика при разборе конкретного бага.
2. **Список сайтов для проверки** — начинаем с «золотого списка» (см. ниже).
3. **Эталонные данные** — результаты ручного осмотра того же сайта (таблица в `tests/fixtures/golden_set_v1.ods`).
4. **Шаблон для записи расхождений** — см. Часть 3.
5. **Браузер с открытым сайтом** — для визуальной перепроверки спорных моментов прямо в ходе тестирования.

### Золотой список сайтов (ниша: онлайн-образование)

| Сайт | Статус эталона | Приоритет |
|------|----------------|-----------|
| el-ed.ru | ✅ Полный (42 пункта) | Первый |
| foxford.ru | Частичный | Второй |
| interneturok.ru | Частичный | Второй |
| webium.ru | Нет данных | Третий |
| tutoronline.ru | Нет данных | Третий |
| egeturbo.ru | Нет данных | Третий |

**Начинай с el-ed.ru** — по нему есть полные эталонные данные ручного осмотра. Расхождения на этом сайте мы уже отчасти знаем, и новые данные сразу сравниваются с историей.

## Часть 2. Как проводить один тестовый прогон

### Шаг 1. Запустить сканирование
1. Открой интерфейс сканера (`/check`).
2. Введи URL сайта (только домен, например `el-ed.ru`).
3. Дождись результата — это занимает от 30 секунд до 3 минут.
4. Зафиксируй время начала и время получения результата (нужно для оценки производительности).

### Шаг 2. Записать общие данные прогона

Сразу запиши в свою таблицу (см. шаблон в Части 3):

| Поле | Что записать |
|------|-------------|
| Дата и время | Когда запускал |
| Сайт | Полный URL |
| Score сканера | Процент из результата |
| Время сканирования | Сколько секунд/минут |
| Модель / версия | Если видно в интерфейсе |
| Примечания | Всё необычное — ошибки, зависания, пустой результат |

### Шаг 3. Сравнить с эталоном по каждому пункту

Открой рядом результат сканера и эталонную таблицу. Пройди по каждому пункту:

```
Для каждого пункта чеклиста:
├── Сканер: PASS / FAIL / не проверялось
├── Эталон (ручной осмотр): ✅ ок / ❌ нарушение / нет данных
└── Сравнение:
    ├── Совпадает → пишем ✓, идём дальше
    ├── Расходятся → пишем РАСХОЖДЕНИЕ и заполняем карточку (Часть 3)
    └── Один из них пустой → пишем НУЖНА ПРОВЕРКА
```

### Шаг 4. Визуальная перепроверка при расхождениях

Если сканер и эталон не совпадают — **не верь ни одному из них на слово**. Открой сам сайт в браузере и проверь глазами: есть ли там то, о чём спорят. Это важно: иногда сайт изменился с момента ручного осмотра.

## Часть 3. Как фиксировать расхождения

Это самая важная часть руководства. Правильно зафиксированное расхождение — это уже половина решения.

### Классификация расхождений

Прежде чем заполнять карточку, определи тип:

| Тип | Что случилось | Серьёзность |
|-----|---------------|-------------|
| **False negative** | Сканер: PASS, эталон: ❌, глаза: нарушение есть | 🔴 Критично |
| **False positive** | Сканер: FAIL, эталон: ✅, глаза: нарушения нет | 🟡 Важно |
| **Сайт изменился** | Сканер и эталон расходятся, но глаза говорят, что сайт другой | ⚪ Не баг сканера |
| **Верно, но не то** | Сканер нашёл что-то, но не то нарушение | 🟡 Важно |
| **Новая находка** | Сканер нашёл нарушение, которого нет в эталоне | 🟢 Проверить |

### Карточка расхождения

Заполняй одну карточку на каждое расхождение. Можно в текстовый файл, в таблицу — как удобнее.

```
=== РАСХОЖДЕНИЕ ===

Дата: ГГГГ-ММ-ДД
Сайт: https://...
Пункт чеклиста: [название пункта, например POLICY_008 или «Правовые основания обработки»]
Тип расхождения: [false negative / false positive / сайт изменился / верно но не то / новая находка]
Серьёзность: [критично / важно / минор]

Что сказал сканер: [PASS / FAIL / не проверялось] — [цитата из результата если есть]

Что показал ручной осмотр: [есть / нет / не смотрели]

Что видно глазами сейчас: [опиши что нашёл на сайте, с URL страницы]

Гипотеза о причине: [необязательно — если понимаешь, почему так]

Дополнительно: [скриншот, ссылка на конкретный элемент на странице]
```

### Пример заполненной карточки

```
=== РАСХОЖДЕНИЕ ===

Дата: 2026-04-18
Сайт: https://foxford.ru
Пункт чеклиста: COOKIE_002 «Кнопка Отклонить в баннере»
Тип расхождения: false negative
Серьёзность: важно

Что сказал сканер: PASS — «Кнопка отклонить обнаружена в баннере»

Что показал ручной осмотр: ❌ — кнопки «Отклонить» нет

Что видно глазами сейчас: На сайте cookie-баннер с одной кнопкой «Принять».
Кнопки отклонить нет вообще. URL: https://foxford.ru (главная)

Гипотеза о причине: Сканер, возможно, ошибочно принял другую кнопку за «Отклонить»

Дополнительно: [скриншот баннера прикреплён]
```

## Часть 4. Что делать после тестовой сессии

### Сводная таблица сессии

По итогам каждой сессии заполни сводку — одна строка на сайт:

| Сайт | Дата | Score | Пунктов совпало | False negative | False positive | Сайт изменился | Новые находки |
|------|------|-------|-----------------|----------------|----------------|----------------|---------------|
| el-ed.ru | 2026-04-18 | 64% | 32/42 | 2 | 1 | 0 | 1 |

### Как передать результаты команде

1. Сводную таблицу сессии — в общий чат или на почту.
2. Карточки расхождений — в папку `docs/test_reports/ДАТА_САЙТ.md` (или аналог).
3. Скриншоты спорных мест — приложить к карточкам.
4. Если нашёл что-то срочное (критичный false negative на очевидном нарушении) — отдельно в чат, не жди следующей сессии.

## Часть 5. Приоритеты при исправлении

Когда команда получает карточки расхождений, она приоритизирует их так:

| Приоритет | Условие |
|-----------|---------|
| 🔴 Блокирующий | False negative по пунктам с severity «critical» или «high» в чеклисте |
| 🟠 Высокий | False negative по «medium», любой false positive на золотых сайтах |
| 🟡 Средний | False positive на сайтах вне золотого списка, «верно но не то» |
| 🟢 Низкий | Косметика, новые находки требующие обсуждения |

## Часть 6. Специальные сценарии тестирования

Помимо сравнения с эталоном, есть несколько дополнительных проверок, которые дают ценную информацию.

### Тест на стабильность (один сайт, два прогона подряд)

Запусти один и тот же сайт дважды с интервалом в 10 минут. Score должен совпасть. Если расходится — это баг нестабильности, важно зафиксировать.

```
Зафиксировать: URL, время первого прогона, score 1, время второго прогона, score 2, расхождение в пунктах
```

### Тест на новый сайт (не из золотого списка)

Попробуй любой сайт онлайн-образования, о котором у нас нет данных. Здесь эталона нет — твоя задача:
1. Запустить сканер
2. Самостоятельно просмотреть сайт по ключевым пунктам (политика, cookie-баннер, формы)
3. Сравнить своё впечатление с результатом
4. Зафиксировать очевидные расхождения

Это называется «exploratory testing» — свободный поиск без сценария.

### Тест на плохой ввод

Проверь, как сканер реагирует на проблемные входные данные:

| Что вводим | Ожидаемое поведение |
|-----------|---------------------|
| Несуществующий домен (abc123xyz.ru) | Понятное сообщение об ошибке |
| Сайт без политики конфиденциальности | score низкий, но не падение |
| Сайт на другом языке | Сканер работает, отмечает отсутствие русской политики |
| URL с опечаткой (https://el-ed.ruu) | Понятная ошибка |
| IP-адрес вместо домена | Понятная ошибка или корректный результат |

Если сканер падает с непонятной ошибкой или зависает — это баг, фиксируй.

### Тест производительности

Отмечай время каждого сканирования. Нормы времени устанавливаются по результатам первых реальных прогонов — у нас пока нет достаточно замеров, чтобы называть конкретные цифры. Для каждого нового сайта запиши время первых 2–3 прогонов и сообщи команде: это поможет установить ориентиры.
Точно фиксируй как проблему:

Сканирование не завершилось через 10 минут (зависание)
Сканер вернул пустой результат без сообщения об ошибке
Сканер упал с непонятной ошибкой

## Часть 7. Что не нужно делать при тестировании

- **Не тестируй на крупных сайтах с WAF** (Wildberries, Ozon, Tinkoff, DNS) — они блокируют автоматические запросы, результат будет непоказательным. Эти сайты — отдельная задача для разработчиков.
- **Не запускай сканер на одном сайте чаще 3 раз в час** — это лишняя нагрузка на сервер и на сайт-цель.
- **Не исправляй ничего в коде сам** — только фиксируй. Исправления — задача разработчиков.
- **Не считай эталон истиной в последней инстанции** — сайты меняются. Если глаза говорят одно, а эталон другое — доверяй глазам и фиксируй оба варианта.

## Часть 8. Быстрая шпаргалка

```
1. Открыть сканер → ввести URL → дождаться результата
2. Записать score и время сканирования
3. Для каждого пункта: сканер vs эталон vs глаза
4. Расхождение → заполнить карточку (тип + что сказал сканер + что видно)
5. Отправить результаты команде
```

**Золотое правило:** лучше зафиксировать расхождение и ошибиться, чем пропустить и не написать. Команда разберётся — для этого она и существует.

## Приложение А. Ключевые пункты для ручного осмотра

Если нет времени на полный осмотр — проверь хотя бы эти 10 пунктов:

1. **Политика конфиденциальности есть** — найди ссылку в футере
2. **Политика на русском языке** — открой и проверь
3. **В политике есть категории данных** — ФИО, email, телефон и т.д.
4. **В политике есть цели обработки** — для чего собирают данные
5. **В политике есть раздел о правах** — как субъект может отозвать согласие
6. **Cookie-баннер появляется** — при первом открытии сайта в чистом браузере
7. **В баннере есть кнопка «Отклонить»** или аналог
8. **В формах есть чекбокс согласия** — не предзаполненный
9. **Рядом с чекбоксом есть ссылка на политику**
10. **Ссылка на политику есть в футере каждой страницы**

## Приложение Б. Пояснение терминов

| Термин | Что означает |
|--------|-------------|
| Score | Итоговый процент соответствия 152-ФЗ по версии сканера |
| PASS | Сканер считает, что этот пункт выполнен (нарушения нет) |
| FAIL | Сканер считает, что этот пункт нарушен |
| False negative | Сканер говорит PASS, реально нарушение есть |
| False positive | Сканер говорит FAIL, реально всё в порядке |
| Эталон | Результаты ручного осмотра сайта экспертом, зафиксированные в golden_set.ods |
| Золотой список | Набор сайтов, по которым у нас есть эталонные данные |
| PDF-политика | Политика конфиденциальности в виде PDF-файла (сложнее для автоматического чтения) |
| WAF | Защита сайта от автоматических запросов (Cloudflare, DDoS-Guard и т.п.) |
