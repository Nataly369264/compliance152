# 📗 ДОКУМЕНТ: Архитектура и логика сканера
# docs_scanner_logic.md — для Claude Code

## Контекст проекта
Сканер проверяет сайты любых операторов персональных данных на соответствие 152-ФЗ
«О персональных данных». На начальном этапе фокус на EdTech, но алгоритм универсален.

**Главный фильтр для любой фичи:**
> «Это помогает определить конкретное нарушение 152-ФЗ?»
> Если нет — не реализовывать.

---

## Архитектура: четыре слоя

```
СЛОЙ 1 — Технический сбор данных
crawler.py + detectors.py
Задача: увидеть сайт максимально полно

СЛОЙ 2 — Юридический анализ
analyzer.py
Задача: определить нарушения по 152-ФЗ
⚠️ Вся юридическая логика только здесь.
Никакой GDPR/CCPA логики — только 152-ФЗ.

СЛОЙ 3 — Отчёт
Строится внутри analyzer.py → ComplianceReport
Статья закона + описание + рекомендация + штраф.
В конце отчёта — блок scan_limitations
и предложение генерации документов.

СЛОЙ 4 — Генерация документов
generator.py → DocumentGenerator
Создаёт готовые документы по 152-ФЗ (~12 типов).
```

> ⚠️ В старой версии документа Слой 3 назывался reporter.py — этого файла нет.
> Отчёт формируется в analyzer.py, документы генерирует generator.py.

## Слой 1: Технический сбор

### Шаг 0 — Tech Fingerprinting

```python
# Цель (часть не реализована — см. бэклог)
if is_js_rendered(html):      # __NEXT_DATA__, пустой <div id="root">
    → режим Playwright         # ⚠️ НЕ РЕАЛИЗОВАН
elif is_pdf_policy(url):      # Content-Type: application/pdf
    → режим pdfplumber         # ⚠️ НЕ РЕАЛИЗОВАН
else:
    → режим httpx + BeautifulSoup  # ✅ реализован, по умолчанию
```

| Маркер в HTML                  | Технология     | Нужен Playwright   |
| ------------------------------ | -------------- | ------------------ |
| `__NEXT_DATA__` или `_next/`   | Next.js        | Да (не реализован) |
| `<div id="root"></div>` пустой | React SPA      | Да (не реализован) |
| `<div id="app"></div>` пустой  | Vue SPA        | Да (не реализован) |
| wp-content/                    | WordPress      | Нет                |
| meta generator="Tilda"         | Tilda          | Нет                |
| Контент прямо в `<body>`       | Статичный HTML | Нет                |

Для определения технологий (запланировано):
`wappalyzer-next` (Python, open-source) — без внешних API-запросов.

### Внешние API (запланировано, не реализовано)
- **urlscan.io** — получение трекеров через реальный браузер.
  Бесплатный тариф: 100 запросов/день. Возвращает: технологии + внешние домены + скриншот.

### Антибот-защита
Если сайт вернул 403/429 (DDoS-Guard, Cloudflare) — фиксировать в `scan_limitations`,
не как баг сканера.
> ⚠️ Сейчас попадает в `errors[]`. Нужна явная обработка в бэклоге.

### Что собирает crawler.py (реализовано)
- Обходит до 50 страниц, приоритет — страницам с политикой ПДн
- Формы: поля, чекбоксы согласия, ссылки на политику
- Внешние ресурсы: JS-скрипты, CSS, шрифты, iframe, пиксели (1×1 img)
- Маркировка запрещённых сервисов через `knowledge_base`
- Cookie-баннер (определяется один раз при первом обнаружении)
- Флаг `analytics_before_consent` — аналитика до согласия
- Fallback-пути политики (20 вариантов: `/privacy`, `/personal-data` и др.)
- SSL-проверка

### Что фиксировать в scan_limitations
Обязательно добавлять в отчёт, если:
- Сайт на JS-рендеринге (Playwright недоступен)
- Политика ПДн в PDF (pdfplumber недоступен)
- Сайт вернул 403/429 (антибот-защита)
- Формы найдены только за авторизацией
- SPA-формы (React/Vue) — checkbox-атрибуты могут не совпадать с HTML-снимком

## Слой 2: Юридический анализ

### Реестр нарушений

Каждое нарушение содержит:
- Уникальный код
- Severity: CRITICAL / HIGH / MEDIUM / LOW
- Статью 152-ФЗ
- Описание понятным языком (не юридическим)
- Рекомендацию
- Диапазон штрафа (для CRITICAL/HIGH)

✅ **НЕ ВКЛЮЧАТЬ — не затрагивают ПДн:**
jQuery, Hammer.js, Google Font API*, jsDelivr, cdnjs, DDoS-Guard, Open Graph, Cloudflare

> *Google Font API — не включать в список трекеров, но фиксировать как TECH_002

### LLM-анализ (реализовано в analyzer.py)
- Глубокий анализ текста политики через Claude (`_analyze_policy_with_llm`)
- Перед анализом — сбор актуального правового контекста из интернета (`web_verification`)
- Результат включается в `ComplianceReport.llm_analysis`

### Оценка штрафов (реализовано)
- Для CRITICAL/HIGH нарушений — автоматический расчёт из `knowledge_base/fine_schedule`
- `FineEstimate` содержит `min_total`, `max_total`, `breakdown` по нарушениям

## Слой 3: Отчёт (ComplianceReport)

```
1. overall_score (0–100) + risk_level (CRITICAL / HIGH / MEDIUM / LOW)
2. Статистика: total_checks / passed / failed / warnings
3. violations — нарушения, сгруппированы по severity
4. checklist — все проверки с результатом
5. fine_estimate — оценка штрафов с разбивкой
6. llm_analysis — глубокий LLM-анализ текста политики
7. summary — LLM-резюме (топ нарушений + рекомендации)
8. scan_limitations — что не смогли проверить
```

## Слой 4: Генерация документов (generator.py)

`DocumentGenerator` создаёт готовые документы по 152-ФЗ на основе данных организации.

**Процесс генерации каждого документа:**
1. Загрузить шаблон из `knowledge_base/templates/`
2. Загрузить правовые обновления из `knowledge_base` (статический JSON)
3. Собрать актуальный веб-контекст (поиск + LLM-суммаризация, кэшируется)
4. Вызвать Claude с данными организации + шаблон + правовой контекст
5. Fallback: заполнить шаблон напрямую (без LLM, с предупреждением)

**PUBLIC_DOCUMENTS** (для сайта):
- `privacy_policy` — Политика обработки ПДн
- `consent_form` — Текст согласия для форм
- `cookie_policy` — Cookie-политика

**FULL_PACKAGE** (~12 документов):
- + Приказ о назначении ответственного за ПДн
- + Регламент обработки ПДн
- + Соглашение о неразглашении
- + Уведомление в РКН
- + Оценка вреда субъектам ПДн
- + Инструкция при инцидентах
- + и др.

## Правила разработки

**Никогда:**
- Не писать скрипты под конкретный сайт — только универсальный код
- Не использовать GDPR/CCPA логику — только 152-ФЗ
- Не переходить к следующему этапу без коммита и подтверждения

**Всегда:**
- После каждого изменения: `pytest tests/ -q`
- Сохранять SESSION_NOTES с описанием что сделано
- Коммит с понятным сообщением после каждого этапа
- Отчёты сохранять с версией: `data/scan_[домен]_v[N].txt`

**При тестировании:**
- Проверить через `git stash` — баг был до правок или после?
- После фикса — прогнать ранее проверенные сайты (регрессия)
